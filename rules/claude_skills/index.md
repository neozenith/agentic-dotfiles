# Claude Skills & Rules — Structure Contract (root)

This is the **root** of the skill/rule documentation family. It governs the one
invariant that applies to *every* file under `.claude/skills/**` and
`.claude/rules/**`: **size**. Its children specialise:

| Sibling | Owns |
|---------|------|
| [`docs.md`](docs.md) | The **documentation** contract — the `SKILL.md` / `README.md` / `CLAUDE.md` trio, diagram gates, the docs done-when checklist. |
| [`scripts.md`](scripts.md) | The **code** contract — the Makefile `fix`/`ci` loop, tests, coverage, per-language tooling. |
| [`../agnostic_examples.md`](../agnostic_examples.md) | The cross-cutting **brand-agnostic** rule that all three must honour. |

Start here, then descend to the sibling that matches your change. Each sibling
links back up to this root, so the family is navigable from any entry point.

## The 500-line invariant

**No skill file and no rule file may exceed 500 lines.**

This applies to everything in scope: `SKILL.md`, `README.md`, `CLAUDE.md`,
`resources/*.md`, every `.claude/rules/**` file (including this one), and any
prose doc the family owns. It is a hard ceiling, not a guideline — a file at 501
lines is **not done**.

Why 500: a file that fits in one read is a file an agent loads, reasons about,
and edits as a single unit. Past ~500 lines the always-loaded context cost rises,
diffs get noisy, and the file starts trying to be two documents at once. The
ceiling forces the split *before* the file becomes unnavigable.

> This invariant is about **hand-maintained documentation and rule prose** — the
> human/agent-readable surface. Three things are **out of scope** because the
> ceiling can't apply meaningfully to them:
> - **Code source files** (`*.py`, `*.ts`) — governed by their own quality gates
>   in [`scripts.md`](scripts.md) (coverage, lint, type).
> - **Test fixtures** — files that exist to be *fed to a script* (e.g. a lint
>   fixture with expected output baked in, wired into the Makefile). Splitting one
>   changes the script's input and breaks the test it backs.
> - **Generated artifacts** — files produced by a `docs`/private-helper target
>   (e.g. a `<!--TOC-->`-populated gallery). Hand-splitting them is overwritten on
>   the next regen; fix the generator, not the output.
>
> When in doubt, check whether the file is referenced as an input/output in a
> Makefile or generated by a helper script before treating it as splittable prose.

## Self-balancing tree structure

When a file approaches the ceiling, do **not** trim it down to fit by deleting
substance. Instead, **rebalance the tree**: split the overflowing node into a
parent index plus cohesive children, the same way a B-tree splits a full node.

Treat the family as a tree of nodes (files), each held under 500 lines:

```
claude_skills/index.md           ← root: the size invariant + family map
├── claude_skills/docs.md        ← child: documentation contract
│   └── (future) resources split if it overflows
├── claude_skills/scripts.md     ← child: code contract
└── agnostic_examples.md         ← sibling rule: cross-cutting constraint (one level up)
```

### How to split a node

1. **Find the natural seam.** Identify the most self-contained subtopic — a
   section that a reader could follow without the surrounding material. That
   becomes the child.
2. **Promote the child to its own file.** Give it a focused title and its own
   `paths:` frontmatter if it should activate independently.
3. **Leave a router stub in the parent.** Replace the moved section with a
   one-to-three-line summary plus a link down to the child. The parent keeps the
   *map*, the child keeps the *depth*.
4. **Link in all three directions** so the tree stays navigable:
   - **Down**: parent → child (the router stub's link).
   - **Up**: child → parent (a "this is part of the … family, see X" line).
   - **Lateral**: child ↔ sibling where topics genuinely cross-reference.
5. **Rebalance, don't just append.** If two siblings have drifted so one is near
   500 lines and another is sparse, move a section across rather than spawning a
   third shallow file. The goal is *balanced* nodes, not maximal fan-out.

### Balance heuristics

- A parent/index node should stay **small** — it routes, it does not duplicate
  the depth of its children. If the index itself nears 500 lines, the family has
  too many topics under one root; introduce an intermediate grouping node.
- A child should be **cohesive**: one audience, one concern. If a child needs
  two unrelated halves, it is itself a split candidate.
- Never duplicate content across nodes to "make each readable standalone."
  Duplication is the failure mode the tree exists to prevent — link instead.
- A `[[link]]` or relative-path link to a node that does not exist yet is a
  *marker for a split worth making later*, not an error.

## Done-when checklist

```text
[ ] Every touched skill/rule file is ≤ 500 lines.
[ ] Any file that would exceed 500 was split at a natural seam, not trimmed of substance.
[ ] The split left a small router stub in the parent + a focused child.
[ ] Up / down / lateral links are present so the node is reachable from the family root.
[ ] No content was duplicated across nodes to dodge the ceiling — links used instead.
```
